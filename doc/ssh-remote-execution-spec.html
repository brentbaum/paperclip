Paperclip Spec Shift

Run remotely is a venue, not an agent.

Existing agents keep their identity and adapter type. A board-level Run remotely choice creates a persistent remote lease for the issue, reuses the same remote branch and session area across resumes, and ends in a PR.

New default

Per-issue remote lease

One issue gets one remote home: repo, branch, adapter-home, portable memory bundle, and PR metadata.

1 issue → 1 lease
Replacement

Old SSH adapter removed

Remote execution moves to issue policy plus remote targets. It is no longer configured as its own agent adapter type.

Core Model

What changes in the product model

The board assigns work to the same agent as before, but now chooses the execution venue at the issue layer.

Identity stays put

Budgeting, activity, org chart, and capabilities remain attached to the same employee record.

Venue becomes explicit

Add executionMode: default | remote on the issue instead of swapping adapter types.

State persists by issue

Follow-up comments and resumes come back to the same remote lease until completion, reset, or TTL cleanup.

Lifecycle

How a remote issue actually moves

The important change is persistence. The first run provisions a lease; later runs reuse it.

1

Assign issue

The board picks an existing agent and toggles Run remotely.

2

Create lease

Paperclip resolves a target host and creates or reuses a remote lease for the issue.

3

Materialize state

The repo is cloned or reused, the issue branch is checked out, and a portable agent bundle is synced.

4

Run adapter

The normal Claude, Codex, Cursor, or OpenCode adapter runs on the remote host and streams transcript events back.

5

Push and PR

The remote worker pushes the stable issue branch, creates or updates the PR, and keeps the lease alive.

Architecture

Where the remote layer fits

The remote worker becomes a hosting layer for existing adapters rather than a special-purpose SSH adapter identity.

Paperclip server Heartbeat service resolves issue policy, adapter type, and workspace source.
Execution orchestrator Chooses local vs remote and attaches the correct session namespace.
Remote venue
SSH remote runtime Creates leases, syncs portable state, and owns branch + PR lifecycle.
Remote issue workspace repo/, adapter-home/, bundle/, metadata.json
Normal adapter runtime Codex, Claude, Cursor, or OpenCode running remotely with stable session storage.
State Strategy

Copy less, model more

The design avoids shipping the entire local machine state. Only intentional, portable state crosses the boundary.

Portable bundle

Include
Instructions files, selected memory files, task/session metadata, skills payloads, run auth.
Exclude
Whole home directories, arbitrary caches, machine-specific paths, vendor login state by default.
Reason
Smaller transfers, fewer secrets, more deterministic resumes, clearer local-vs-remote behavior.

Decision Matrix

Why this is the preferred structure

It is the cleanest fit for Paperclip’s control-plane model and avoids conflating employee identity with runtime placement.

Recommended

Existing agents + remote leases

  • Correct mental model
  • Persistent per-issue continuity
  • Clean local vs remote separation
  • Natural path to host pools later
Rejected

Dedicated remote adapter type

  • Smaller code delta at first
  • But remote becomes part of agent identity
  • Confusing when issues switch venue
  • Duplicates employees unnecessarily
Rejected default

Copy full working directory

  • Looks simple at first
  • Copies too much state and too many secrets
  • Fragile across absolute paths and auth stores
  • Hard to explain and debug
Delivery

Implementation sequence

The spec now has a clear order: model the issue policy first, then persistence, then remote execution details.

Phase 1

  • Add issue-level executionMode = remote
  • Add remote execution targets
  • Add remote lease records
  • Separate session namespaces by venue

Phase 2

  • Run existing adapters remotely
  • Sync portable state bundles
  • Manage stable remote branches
  • Create or update PRs and surface metadata
